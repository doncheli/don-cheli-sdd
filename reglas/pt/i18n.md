# i18n — Regras de Internacionalização

## Detecção de Idioma

Em cada interação, detectar o idioma configurado pelo usuário:

1. Ler o arquivo de locale do framework em `${FRAMEWORK_HOME}/locale` (contém `es`, `en` ou `pt`)
2. Se não encontrado, verificar `.dc/config.yaml` → `framework.idioma`
3. Se nenhum existir, usar `es` como padrão

## Comportamento por Idioma

Uma vez determinado o locale:

| Locale | Comunicação | Código | Commits/PRs | Docs |
|--------|------------|------|-------------|------|
| `es` | Español | English | Español | Español |
| `en` | English | English | English | English |
| `pt` | Português | English | Português | Português |

**Código (variáveis, funções, comentários) é SEMPRE em inglês, independente do locale.**

## Arquivos de Locale

As strings de locale do framework estão em `locales/{es,en,pt}.json`. Usá-las para:
- Nomes de fases, rótulos de status, veredictos
- Mensagens do instalador
- Templates de saída de comandos
- Dimensões de revisão, rótulos de severidade
- Nomes do ciclo TDD, leis de ferro

## Como Aplicar

Ao gerar saídas (estado.md, plan.md, hallazgos.md, propuesta.md, etc.):
- Usar a terminologia do locale para cabeçalhos, rótulos e textos de status
- Manter termos técnicos sem tradução (ex: TDD, OWASP, API, WebSocket)
- As palavras-chave do Gherkin seguem a spec i18n do Gherkin:
  - `es`: Dado/Quando/Então/E
  - `en`: Given/When/Then/And
  - `pt`: Dado/Quando/Então/E

## Alterando o Idioma

O usuário pode alterar o idioma a qualquer momento:
1. Editando diretamente `${FRAMEWORK_HOME}/locale`
2. Re-executando `scripts/instalar.sh` (reapresenta a seleção de idioma)
3. Definindo `framework.idioma` em `.dc/config.yaml`

## Convenção por Camada de Definição

O framework define skills em 3 camadas com convenções de idioma distintas:

| Camada | Localização | Campos | Idioma do conteúdo | Audiência |
|--------|------------|--------|-------------------|-----------|
| **HABILIDAD.md** | `habilidades/*/HABILIDAD.md` | `nombre:`, `descripcion:` | Espanhol | Documentação interna |
| **Comando** | `comandos/especdev/*.md` | `description:`, `i18n:` | Espanhol + keywords bilíngues | Claude Code CLI (routing) |
| **SKILL.md** | `.agent/skills/doncheli-*/SKILL.md` | `name:`, `description:` | Inglês | Anthropic Skills 2.0 (máquina) |

### Por que cada camada usa um idioma diferente

- **HABILIDAD.md** usa espanhol porque é documentação interna do framework, consumida por desenvolvedores hispanófonos. Campos em espanhol (`nombre:`, `descripcion:`) por consistência com o locale principal.
- **Comando** usa espanhol para a descrição principal (visível ao usuário) com keywords bilíngues no mesmo campo `description:` para melhorar o matching de intenção (ex: `"deuda técnica", "technical debt", "code smells"`).
- **SKILL.md** usa inglês porque é o formato exigido pela especificação Anthropic Skills 2.0. Os campos `name:` e `description:` são consumidos pelo Claude Code para skill activation.

### Nomenclatura de diretórios SKILL.md

Os diretórios de skills usam o nome do comando traduzido ao inglês com prefixo `doncheli-`:

| Comando (espanhol) | Diretório SKILL.md (inglês) |
|--------------------|-----------------------------|
| `/dc:implementar` | `doncheli-implement/` |
| `/dc:estimar` | `doncheli-estimate/` |
| `/dc:especificar` | `doncheli-spec/` |
| `/dc:desglosar` | `doncheli-breakdown/` |

### Validação

- CI (`validar.yml`) verifica que SKILL.md tenha `name:` e `description:` no frontmatter
- CI verifica que comandos tenham `description:` no frontmatter
- Para HABILIDAD.md: os campos devem ser `nombre:` e `descripcion:` (espanhol)
