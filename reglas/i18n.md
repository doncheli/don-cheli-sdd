# i18n — Internationalization Rules

## Language Detection

On every interaction, detect the user's configured language:

1. Read the framework locale file at `${FRAMEWORK_HOME}/locale` (contains `es`, `en`, or `pt`)
2. If not found, check `.dc/config.yaml` → `framework.idioma`
3. If neither exists, default to `es`

## Language Behavior

Once the locale is determined:

| Locale | Communication | Code | Commits/PRs | Docs |
|--------|--------------|------|-------------|------|
| `es` | Español | English | Español | Español |
| `en` | English | English | English | English |
| `pt` | Português | English | Português | Português |

**Code (variables, functions, comments) is ALWAYS in English regardless of locale.**

## Locale Files

Framework locale strings are in `locales/{es,en,pt}.json`. Use them for:
- Phase names, status labels, verdicts
- Installer messages
- Command output templates
- Review dimensions, severity labels
- TDD cycle names, iron laws

## How to Apply

When generating output (estado.md, plan.md, hallazgos.md, propuesta.md, etc.):
- Use the locale's terminology for headers, labels, and status text
- Keep technical terms (e.g., TDD, OWASP, API, WebSocket) untranslated
- Gherkin keywords follow the Gherkin i18n spec:
  - `es`: Dado/Cuando/Entonces/Y
  - `en`: Given/When/Then/And
  - `pt`: Dado/Quando/Então/E

## Changing Language

The user can change the language at any time by:
1. Editing `${FRAMEWORK_HOME}/locale` directly
2. Re-running `scripts/instalar.sh` (re-prompts language selection)
3. Setting `framework.idioma` in `.dc/config.yaml`

## Convención por Capa de Definición

El framework define skills en 3 capas con convenciones de idioma distintas:

| Capa | Ubicación | Campos | Idioma contenido | Audiencia |
|------|-----------|--------|-----------------|-----------|
| **HABILIDAD.md** | `habilidades/*/HABILIDAD.md` | `nombre:`, `descripcion:` | Español | Documentación interna |
| **Comando** | `comandos/especdev/*.md` | `description:`, `i18n:` | Español + keywords bilingües | Claude Code CLI (routing) |
| **SKILL.md** | `.agent/skills/doncheli-*/SKILL.md` | `name:`, `description:` | Inglés | Anthropic Skills 2.0 (máquina) |

### Por qué cada capa tiene un idioma diferente

- **HABILIDAD.md** usa español porque es documentación interna del framework, consumida por desarrolladores hispanoparlantes. Campos en español (`nombre:`, `descripcion:`) por consistencia con el locale principal.
- **Comando** usa español para la descripción principal (visible al usuario) con keywords bilingües en la misma `description:` para mejorar el matching de intención (ej: `"deuda técnica", "technical debt", "code smells"`).
- **SKILL.md** usa inglés porque es el formato requerido por la especificación Anthropic Skills 2.0. Los campos `name:` y `description:` son consumidos por Claude Code para skill activation.

### Naming de directorios SKILL.md

Los directorios de skills usan el nombre del comando traducido al inglés con prefijo `doncheli-`:

| Comando (español) | Directorio SKILL.md (inglés) |
|-------------------|------------------------------|
| `/dc:implementar` | `doncheli-implement/` |
| `/dc:estimar` | `doncheli-estimate/` |
| `/dc:especificar` | `doncheli-spec/` |
| `/dc:desglosar` | `doncheli-breakdown/` |

### Validación

- CI (`validar.yml`) verifica que SKILL.md tenga `name:` y `description:` en frontmatter
- CI verifica que comandos tengan `description:` en frontmatter
- Para HABILIDAD.md: los campos deben ser `nombre:` y `descripcion:` (español)
