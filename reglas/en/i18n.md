# i18n — Internationalization Rules

## Language Detection

On every interaction, detect the user's configured language:

1. Read the framework locale file at `${FRAMEWORK_HOME}/locale` (contains `es`, `en`, or `pt`)
2. If not found, check `.dc/config.yaml` → `framework.idioma`
3. If neither exists, default to `es`

## Language Behavior

Once the locale is determined:

| Locale | Communication | Code | Commits/PRs | Docs |
|--------|--------------|------|-------------|------|
| `es` | Español | English | Español | Español |
| `en` | English | English | English | English |
| `pt` | Português | English | Português | Português |

**Code (variables, functions, comments) is ALWAYS in English regardless of locale.**

## Locale Files

Framework locale strings are in `locales/{es,en,pt}.json`. Use them for:
- Phase names, status labels, verdicts
- Installer messages
- Command output templates
- Review dimensions, severity labels
- TDD cycle names, iron laws

## How to Apply

When generating output (estado.md, plan.md, hallazgos.md, propuesta.md, etc.):
- Use the locale's terminology for headers, labels, and status text
- Keep technical terms (e.g., TDD, OWASP, API, WebSocket) untranslated
- Gherkin keywords follow the Gherkin i18n spec:
  - `es`: Dado/Cuando/Entonces/Y
  - `en`: Given/When/Then/And
  - `pt`: Dado/Quando/Então/E

## Changing Language

The user can change the language at any time by:
1. Editing `${FRAMEWORK_HOME}/locale` directly
2. Re-running `scripts/instalar.sh` (re-prompts language selection)
3. Setting `framework.idioma` in `.dc/config.yaml`

## Convention per Definition Layer

The framework defines skills across 3 layers with distinct language conventions:

| Layer | Location | Fields | Content Language | Audience |
|-------|----------|--------|-----------------|----------|
| **HABILIDAD.md** | `habilidades/*/HABILIDAD.md` | `nombre:`, `descripcion:` | Spanish | Internal documentation |
| **Command** | `comandos/especdev/*.md` | `description:`, `i18n:` | Spanish + bilingual keywords | Claude Code CLI (routing) |
| **SKILL.md** | `.agent/skills/doncheli-*/SKILL.md` | `name:`, `description:` | English | Anthropic Skills 2.0 (machine) |

### Why each layer uses a different language

- **HABILIDAD.md** uses Spanish because it is internal framework documentation consumed by Spanish-speaking developers. Fields use Spanish (`nombre:`, `descripcion:`) for consistency with the primary locale.
- **Command** uses Spanish for the main description (user-facing) with bilingual keywords in the same `description:` field to improve intent matching (e.g., `"deuda técnica", "technical debt", "code smells"`).
- **SKILL.md** uses English because it follows the Anthropic Skills 2.0 specification. The `name:` and `description:` fields are consumed by Claude Code for skill activation.

### SKILL.md directory naming

Skill directories use the command name translated to English with `doncheli-` prefix:

| Command (Spanish) | SKILL.md Directory (English) |
|-------------------|------------------------------|
| `/dc:implementar` | `doncheli-implement/` |
| `/dc:estimar` | `doncheli-estimate/` |
| `/dc:especificar` | `doncheli-spec/` |
| `/dc:desglosar` | `doncheli-breakdown/` |

### Validation

- CI (`validar.yml`) verifies SKILL.md has `name:` and `description:` in frontmatter
- CI verifies commands have `description:` in frontmatter
- For HABILIDAD.md: fields must be `nombre:` and `descripcion:` (Spanish)
